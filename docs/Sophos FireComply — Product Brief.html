FireComply then matches the SNMP-retrieved serial against the Sophos Central API inventory to auto-link the firewall — pulling in hostname, model, firmware version, fleet tags, and tenant context without manual data entry. This auto-linking is only available when the Connector Agent has SNMP configured for the firewall.

The Sophos XML config export does not include the serial number. When the Connector Agent has SNMP configured, it retrieves the serial and attaches it to the upload. SNMP Discovery

Connector Agent Serial Retrieval

The Connector Agent queries each firewall via SNMP to retrieve its serial number before pulling the full XML API config export. This serial is attached to every config upload, enabling Central API auto-linking. Without SNMP configured on the agent, the auto-link cannot occur.

How it works: The Connector Agent runs an SNMP GET against each firewall's SNMPv2-MIB::sysDescr OID to capture the serial number. On upload, FireComply calls the Sophos Central /common/v1/firewalls endpoint, matches the serial, and auto-populates the firewall's metadata. Manual HTML/XML uploads do not include the serial, so auto-linking requires the Connector Agent with SNMP enabled.

Behind the Reports

Prompt Engineering & AI Configuration

Getting AI-generated firewall reports to produce accurate, audit-grade output required extensive prompt engineering. FireComply uses a layered system-prompt architecture with domain-specific rules, dynamic context injection, and a data pipeline that anonymises, prunes, and shapes the payload before it reaches the model.

Prompt Assembly Pipeline
Base Prompt + Shared Rules + Dynamic Context = System Prompt
Config JSON → Omit 120+ sections → Prune empty → Anonymise = User Message
→ Gemini / Claude / ChatGPT  (temp 0.1, reasoning: low, max 32k–65k tokens) → SSE stream → de-anonymise → render
Four Prompt Layers
LayerPurposeSize
Base Prompt Role definition, output format (Markdown), table rules, section ordering, severity labels, column handling. Three variants: Technical Handover (~3,500 words), Executive Summary (~500 words), Compliance (~1,200 words). ~5,200 words total
Shared Assessment Rules Domain-specific Sophos rules appended to every non-chat prompt — web filter scope, VPN profiles, wireless, external log forwarding, API service accounts, SSL/TLS DPI engine semantics, omitted sections. ~300 words
Dynamic Context Generated per-request: customer name, environment type, country, selected compliance frameworks, per-firewall scope (multi-jurisdiction), Central enrichment data, web filter compliance mode, jurisdictional notes, report date. Variable
User Message Framing instruction ("Produce a comprehensive Compliance Readiness Report…") plus anonymised, pruned configuration JSON payload. 5k–100k tokens
Example: Shared Assessment Rules (verbatim from codebase)
// Appended to every system prompt
- Web filter scope: Only flag missing web filtering for rules whose
  Destination Zone is WAN and Service is HTTP, HTTPS, or ANY.
  Do not flag LAN-to-LAN, VPN-to-LAN, or non-web service rules.

- VPN Profiles: Only flag weaknesses when the profile is
  actively referenced by a VPN IPSec Connection. Unused = compliant.

- API / Monitoring Service Accounts: Accounts with "Read-Only
  Administrator" profile cannot use interactive MFA. Document
  compensating controls (IP restriction, read-only, audit logging).
  With compensating controls = accepted exception under ISO 27001
  A.9.4.2, Cyber Essentials, NIST 800-53 IA-2, and PCI DSS 8.3.

- SSL/TLS (DPI engine): These settings are for the DPI/inspection
  engine — not a TLS-terminating service. Do not recommend
  "Enforce TLS 1.2+ to comply with NCSC Guidelines" — NCSC applies
  to services presenting TLS to end users, not the firewall's
  minimum decryption version.
Why this matters: Without these rules, the AI consistently produced false positives — flagging VPN profiles that weren't in use, recommending NCSC TLS compliance for an inspection engine, and treating Central-managed firewalls as missing syslog forwarding. Each rule was added to fix a specific class of inaccurate output discovered during testing.

Data Pipeline — Before the AI Sees Anything
Step 1

Section Omission

120+ low-value config sections (DHCP, DNS routes, QoS, email scanning, schedules, services, etc.) are stripped client-side before sending — saving 30–50% of input tokens on a typical config.

Step 2

Payload Pruning

Empty strings, empty arrays, and empty objects are recursively removed from the JSON tree. This eliminates noise and keeps the model focused on actual configuration data.

Step 3

Anonymisation

Hostnames, IP addresses, and identifiers are replaced with tokens before leaving the browser. The AI generates the report using anonymised data; a streaming de-anonymiser restores real values in the response.

Step 4

Dynamic Context Injection

Customer name, environment type, country, selected compliance frameworks, per-firewall jurisdiction, Central API enrichment data, and web filter compliance mode are appended to the system prompt at request time.

SE Engineer Notes — Template-Based Generation
The SE Health Check "Engineer Notes" use a different approach — deterministic template generation rather than LLM inference. The code computes the best-practice score, counts findings by severity, identifies the top failed checks, and assembles natural-language paragraphs using conditional templates. This ensures the notes are always numerically accurate and consistent with the score displayed on screen — no hallucination risk. The conversational tone ("I reviewed…", "The main gaps I spotted…", "If I were to pick…") is baked into the template strings, not generated by a model.
Model tuning: Reports use temperature: 0.1 for near-deterministic output, reasoning_effort: low to balance quality and cost, and max_tokens: 32,768–65,536 to accommodate 30-page reports. A fallback model (Gemini 2.0 Flash) activates automatically if the primary model rate-limits or errors.

Production Infrastructure
Resilience

Model Fallback Cascade

Primary model (Gemini 2.5 Flash) → 12s backoff retry → fallback model (Gemini 2.0 Flash). Google tracks quotas per model, so a model switch often succeeds when the primary is rate-limited. Separate retry logic for 429, 402, and 403 errors.

Rate Limiting

Per-User Rolling Window

Chat and report completions are tracked in separate DB-backed buckets with a configurable per-minute cap. Prevents runaway API spend in multi-tenant deployments without per-customer AI keys. Defaults off — Google billing/quotas apply.

Streaming

SSE with Time Budget

Immediate empty SSE chunk on connection for instant UI feedback. Configurable stream budget (default 138s) with graceful timeout — sends an EDGE_TIME_LIMIT event and explicit [DONE] so the client never hangs. Reports stream chunk-by-chunk from the model.

Observability

Token Usage Tracking

Parses both OpenAI-style usage and Gemini-native usage_metadata from the final SSE chunk. Logs prompt tokens, completion tokens, and total to gemini_usage table per user, per mode, per model. Warns when a single request exceeds 250k tokens (free-tier TPM).

Code: Model Routing & Fallback (verbatim from index.ts)
const reportModel = env("GEMINI_REPORT_MODEL") || "gemini-2.5-flash";
const chatModel  = env("GEMINI_CHAT_MODEL")  || "gemini-2.5-flash-lite";
const fallback  = env("GEMINI_FALLBACK_MODEL") || "gemini-2.0-flash";

// Primary → retry with 12s backoff → switch to fallback model
let response = await doRequest(primaryModel);
if (response.status === 429) {
  await sleep(12_000); // backoff
  response = await doRequest(primaryModel);
}
if (response.status === 429 && fallback !== primary) {
  usedModel = fallbackModel;
  response = await doRequest(fallbackModel); // different quota bucket
}

FireComply Connector Agent

A lightweight Electron agent that runs on-premises and automatically collects firewall configuration exports via the Sophos XML API. Pushes configs to your FireComply workspace on a schedule — no manual uploads required.

Main dashboard — connected with live heartbeat, firewall version and score at a glance

Settings — per-firewall config with host, port, API credentials, SNMP, and connection testing

Live logs — XML API config export of 163 entities alongside SNMP serial number retrieval for Central auto-linking

How It Works
1

Create API User

On each Sophos Firewall, create a read-only API profile and user with API access enabled

2

Install Agent

Download the Electron agent for Windows, macOS, or Linux. Register with your FireComply workspace

3

Add Firewalls

Enter firewall IP/hostname, API credentials, and SNMP community string. The agent tests both connections

4

SNMP Serial

The agent queries the firewall via SNMP to retrieve the serial number for automatic Central API linking

5

Auto Sync & Link

On schedule, the agent pulls config exports, attaches the SNMP serial, and uploads to your workspace where Central auto-links the firewall

SNMP serial number → Central auto-linking: Each scan begins with an SNMP GET to retrieve the firewall's serial number. When the config reaches FireComply, the serial is matched against the Sophos Central /common/v1/firewalls inventory. The firewall's hostname, model, firmware, subscription, and group membership are auto-populated — eliminating manual data entry and ensuring assessment metadata stays accurate across recurring scans.
