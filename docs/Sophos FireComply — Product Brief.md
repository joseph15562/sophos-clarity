## AI Reports & Deliverables

### Four Report Types, One Click

FireComply generates four distinct report types — three AI-powered MSP reports combined into a single document, and a dedicated SE Health Check report. Each is exportable as PDF, Word, HTML, or shareable via link.

#### MSP Report 1 of 3 — Full Technical Handover

Deep-dive per-firewall analysis: every rule, interface, zone, NAT entry, auth server, IPS engine, wireless network, SSL/TLS inspection setting, and admin config — with section-by-section security recommendations and remediation steps. Typically 15–20 pages per firewall.

#### MSP Report 2 of 3 — Executive Summary

Non-technical estate overview: executive narrative, per-firewall summary (key stats, top concerns, DPI status), cross-estate findings, numbered risk matrix with severity ratings, strategic recommendations prioritised by impact, and an appendix with configuration highlights.

#### MSP Report 3 of 3 — Compliance Readiness Report

Framework-mapped evidence tables: security feature gaps per firewall, then control-by-control evidence mapping for selected frameworks (GDPR, Cyber Essentials, NCSC, PCI DSS, ISO 27001, DfE/KCSIE, etc.) with pass/partial/fail status, configuration evidence citations, and remediation impact notes.

SE-focused report: executive summary with Sophos best-practice score and grade, licence tier alignment, provenance and limitations, assessment scope and exclusions (DPI, MDR, DNS, Heartbeat), per-device baseline checklist, full findings table with severity and remediation, and AI-written SE Engineer Notes summarising the review in natural language.

## AI Engineering

### Behind the Reports — Prompt Engineering & AI Configuration

Getting AI-generated firewall reports to produce accurate, audit-grade output required extensive prompt engineering. FireComply uses a layered system-prompt architecture with domain-specific rules, dynamic context injection, and a data pipeline that anonymises, prunes, and shapes the payload before it reaches the model.

#### Prompt Assembly Pipeline

```
Base Prompt  +  Shared Rules  +  Dynamic Context  =  System Prompt
Config JSON  →  Omit 120+ sections  →  Prune empty  →  Anonymise  =  User Message
→ Gemini / Claude / ChatGPT  (temp 0.1, reasoning: low, max 32k–65k tokens) → SSE stream → de-anonymise → render
```

#### Four Prompt Layers

| LAYER                       | PURPOSE                                                                                                                                                                                                                           | SIZE               |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------ |
| **Base Prompt**             | Role definition, output format (Markdown), table rules, section ordering, severity labels, column handling. Three variants: Technical Handover (~3,500 words), Executive Summary (~500 words), Compliance (~1,200 words).         | ~5,200 words total |
| **Shared Assessment Rules** | Domain-specific Sophos rules appended to every non-chat prompt — web filter scope, VPN profiles, wireless, external log forwarding, API service accounts, SSL/TLS DPI engine semantics, omitted sections.                         | ~300 words         |
| **Dynamic Context**         | Generated per-request: customer name, environment type, country, selected compliance frameworks, per-firewall scope (multi-jurisdiction), Central enrichment data, web filter compliance mode, jurisdictional notes, report date. | Variable           |
| **User Message**            | Framing instruction ("Produce a comprehensive Compliance Readiness Report…") plus anonymised, pruned configuration JSON payload.                                                                                                  | 5k–100k tokens     |

#### Example: Shared Assessment Rules (Verbatim From Codebase)

```
// Appended to every system prompt
- Web filter scope: Only flag missing web filtering for rules whose
  Destination Zone is WAN and Service is HTTP, HTTPS, or ANY.
  Do not flag LAN-to-LAN, VPN-to-LAN, or non-web service rules.

- VPN Profiles: Only flag weaknesses when the profile is
  actively referenced by a VPN IPSec Connection. Unused = compliant.

- API / Monitoring Service Accounts: Accounts with "Read-Only
  Administrator" profile cannot use interactive MFA. Document
  compensating controls (IP restriction, read-only, audit logging).
  With compensating controls = accepted exception under ISO 27001
  A.9.4.2, Cyber Essentials, NIST 800-53 IA-2, and PCI DSS 8.3.

- SSL/TLS (DPI engine): These settings are for the DPI/inspection
  engine — not a TLS-terminating service. Do not recommend
  "Enforce TLS 1.2+ to comply with NCSC Guidelines" — NCSC applies
  to services presenting TLS to end users, not the firewall's
  minimum decryption version.
```

Why this matters: Without these rules, the AI consistently produced false positives — flagging VPN profiles that weren't in use, recommending NCSC TLS compliance for an inspection engine, and treating Central-managed firewalls as missing syslog forwarding. Each rule was added to fix a specific class of inaccurate output discovered during testing.

### Prompt Engineering — Continued: Precision Through Constraints

#### Example: Technical Handover — Firewall Rules Instructions

```
// From the 3,500-word Technical Handover system prompt
The table MUST have the rule number (#) as the first column,
then Rule Name, then the rest. Include columns for Web Filter,
Logging, and App Control when data exists.

Do not include: Description, Schedule, Min Destination HB,
Source/Dest Security Heartbeat, Block Quick QUIC, SSL/TLS
Decryption, Min Source HB, Security Features.

Use exactly "Enabled" or "Disabled" (capitalised) for Status,
Logging, and any on/off fields.

If no web filter data for a rule, show "None" — only call it out
in Findings when Destination Zone = WAN and Service = HTTP,
HTTPS, or ANY.

Order for ≤150 rules: Output all rule rows in the table
before writing Summary, Findings, or Best Practice — do not
stop mid-table to write narrative.

Order for >150 rules: Output first 150 rows, then one summary
row ("N more rules; see export"), then continue with analysis.
```

#### Example: Compliance Report — Evidence Mapping Rules

```
// From the 1,200-word Compliance system prompt
When multiple firewalls are assessed, security features must
be enabled across all firewalls and in all relevant areas.
If any single firewall lacks a required feature in any area,
show ⚠ Partial and state which firewall and what is missing.

Section 2 — Control → Evidence Mapping:
Columns: Control ID | Control Description | Status |
Firewall(s) Lacking / What Is Missing | Evidence |
Example of non-compliance | Notes

Status values: Met | ⚠ Partial | Not Met | N/A

Logging / monitoring rule: For any control that relates to
monitoring, logging, or audit trail — fail that control (Not
Met) if config shows one or more rules with logging disabled.
Note which rule(s) and which firewall(s) in Evidence/Notes.
```

#### Data Pipeline — Before the AI Sees Anything

1. **Section Omission** — 120+ low-value config sections (DHCP, DNS routes, QoS, email scanning, schedules, services, etc.) are stripped client-side before sending — saving 30–50% of input tokens on a typical config.
2. **Payload Pruning** — Empty strings, empty arrays, and empty objects are recursively removed from the JSON tree. This eliminates noise and keeps the model focused on actual configuration data.
3. **Anonymisation** — Hostnames, IP addresses, and identifiers are replaced with tokens before leaving the browser. The AI generates the report using anonymised data; a streaming de-anonymiser restores real values in the response.
4. **Dynamic Context Injection** — Customer name, environment type, country, selected compliance frameworks, per-firewall jurisdiction, Central API enrichment data, and web filter compliance mode are appended to the system prompt at request time.

#### SE Engineer Notes — Template-Based Generation

The SE Health Check "Engineer Notes" use a different approach — deterministic template generation rather than LLM inference. The code computes the best-practice score, counts findings by severity, identifies the top failed checks, and assembles natural-language paragraphs using conditional templates. This ensures the notes are always numerically accurate and consistent with the score displayed on screen — no hallucination risk. The conversational tone ("I reviewed…", "The main gaps I spotted…", "If I were to pick…") is baked into the template strings, not generated by a model.

Model tuning: Reports use temperature: 0.1 for near-deterministic output, reasoning_effort: low to balance quality and cost, and max_tokens: 32,768–65,536 to accommodate 30-page reports. A fallback model (Gemini 2.0 Flash) activates automatically if the primary model rate-limits or errors.

## FireComply Connector Agent

### Automated Collection

A lightweight Electron agent that runs on-premises and automatically collects firewall configuration exports via the Sophos XML API. Pushes configs to your FireComply workspace on a schedule — no manual uploads required.

#### How It Works

1. **Create API User** — On each Sophos Firewall, create a read-only API profile and user with API access enabled
2. **Install Agent** — Download the Electron agent for Windows, macOS, or Linux. Register with your FireComply workspace
3. **Add Firewalls** — Enter firewall IP/hostname and API credentials. The agent verifies connectivity and starts monitoring
4. **Automatic Sync** — On schedule, the agent pulls config exports, runs analysis, and pushes results to your workspace

Continuous compliance: When connector agents push fresh configs, Fleet and Mission Control update automatically — scores recalculate, drift detection triggers, and stale-assessment alerts clear. The workspace header shows Connector: 51 agents with real-time status. Agents support heartbeat monitoring, remote scan triggers, and auto-updates.

### Firewall Preparation — Built-In Setup Guide

Before the agent can collect configs, each firewall needs a read-only API user. The connector includes a full setup guide with annotated screenshots walking through every step on the Sophos XGS console.

#### Setup Steps (From the Built-In Guide)

1. **Get API Key** — In FireComply web app, go to Management Panel → Settings → Connector Agents and click Register Agent. Copy the generated API key.
2. **Create Read-Only Profile** — On the Sophos Firewall, navigate to Profiles → Device Access. Create a new profile named "API read only" with read-only permissions.
3. **Create API User** — Add a new administrator (e.g. "firecomply-api") with the read-only profile, enable API access, and set a strong password.
