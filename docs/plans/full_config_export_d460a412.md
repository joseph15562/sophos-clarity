# Full Firewall Config Export via Connector

## Problem

The connector currently exports only 15 of ~130 entity types available from the Sophos XML API. Users must manually export an `Entities.xml` file and run it through a config viewer to see the full firewall configuration. The goal is to have the connector grab everything automatically.

## What changes

### 1. Expand entity list in the connector

**File:** `[firecomply-connector/src/firewall/export-config.ts](firecomply-connector/src/firewall/export-config.ts)`

Replace the hardcoded 10 `CORE_ENTITIES` + 5 `GATED_ENTITIES` with the full list of ~130 entity types extracted from the Entities.xml. The `getEntity()` function already uses `<Get><EntityType/></Get>` which is the same API call format ‚Äî no protocol changes needed.

Sensitive entities to skip or redact:

- `User` (contains passwords)
- `OTPTokens` (contains secrets)
- Entities with password fields will have those fields stripped before transmission

Entities to exclude entirely (large, low-value):

- `CertificateAuthority` (~250 built-in CAs, adds bulk with no compliance value)
- `FormTemplate` (HTML templates)
- `Messages` (UI text)
- `IviewCustomLogo` (binary logo)

This grows the export from ~15 to ~120 entities. With parallel fetching (batches of 5), this should complete in under 60 seconds.

### 2. Include raw config data in the submission payload

**Files:**

- `[firecomply-connector/src/api/submit.ts](firecomply-connector/src/api/submit.ts)` ‚Äî add `raw_config` field to payload (JSON-serialised parsed entities)
- `[firecomply-connector/src/firewall/parse-entities.ts](firecomply-connector/src/firewall/parse-entities.ts)` ‚Äî parse all entities generically into structured JSON
- `[supabase/functions/api/index.ts](supabase/functions/api/index.ts)` ‚Äî store `raw_config` in a new `submissions.raw_config` JSONB column

The raw config will be a JSON object keyed by entity type, with arrays of parsed entity objects. The `ExtractedSections` format (tables, details) is already generated by `parseEntityResults()` ‚Äî this just needs to include all entities, not just the 15 it currently maps.

### 3. Add database column for raw config

New migration: `supabase/migrations/XXXXXX_submission_raw_config.sql`

```sql
ALTER TABLE submissions ADD COLUMN IF NOT EXISTS raw_config jsonb DEFAULT NULL;
```

### 4. Expand the connector analysis

**File:** `[firecomply-connector/src/analysis/analyse-config.ts](firecomply-connector/src/analysis/analyse-config.ts)`

With the new entities available, add analysis checks for:

- `AdminSettings` ‚Äî hostname, admin session timeout
- `OTPSettings` ‚Äî MFA/OTP enforcement
- `DNS` ‚Äî DNS configuration
- `SyslogServers` ‚Äî logging to external SIEM
- `BackupRestore` ‚Äî backup schedule
- `SystemServices` ‚Äî unnecessary services enabled
- `VPNIPSecConnection` / `SSLVPNPolicy` ‚Äî VPN configuration
- `DoSSettings` ‚Äî DoS protection

### 5. Update progress reporting

The connector dashboard and logs should show progress through all ~120 entities (currently shows "14/14"). The existing `onProgress` callback already supports this.

## What does NOT change

- The HTML config viewer in the web app continues to work for manual uploads
- The analysis/scoring pipeline is additive ‚Äî existing checks are untouched
- The API key, heartbeat, and SNMP flows are unchanged

## Estimated impact

- Export time: ~15s currently -> ~45-60s with all entities (sequential) or ~15-20s (batched)
- Payload size: ~50KB currently -> ~200-500KB with raw config (compressed)
- No breaking changes to existing data
